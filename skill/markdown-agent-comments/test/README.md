# Tests And Evals | Markdown Agent Comments

## Overview

This folder answers three different questions:

1. Does the deterministic scanner find the right `@agent` comments?
2. Can an agent using `SKILL.md` process real markdown files correctly?
3. Is it easy for a human to understand what common cases are supported and what gets ignored?

## How It Works

The scanner is tested using pass/fail deterministic automated tests.

The skill is tested using evals. An agent gets markdown files with comments to resolve and writes output into a generated run directory. A deterministic verifier catches protocol regressions, and an LLM judge can score answer quality against reference outputs in `expected/`.

The human demo has two parts: a curated before/after fixture humans can read quickly, and a real LLM demo runner that copies `demo.md` to an ignored run directory and invokes `SKILL.md` against that copy.

## Quick Start

Run the normal test suite:

```sh
bun run test
```

Run tests plus a real LLM demo skill run:

```sh
MDAC_DEMO_AGENT_COMMAND="codex exec --ignore-user-config --ignore-rules --ephemeral --skip-git-repo-check --full-auto -m gpt-5.3-codex-spark" bun run test:review
```

Prepare a skill eval run:

```sh
bun run eval:prepare -- --executor manual --run-id local-smoke
```

Then open the generated prompt:

```sh
skill/markdown-agent-comments/test/fixtures/skill-evals/runs/local-smoke/executor-prompt.md
```

Give only that generated prompt to an agent. Do not paste this README or any `expected/` files into the executor chat. The agent should edit only files in:

```sh
skill/markdown-agent-comments/test/fixtures/skill-evals/runs/local-smoke/actual/
```

Verify the run mechanically:

```sh
bun run eval:verify -- --run local-smoke --write
```

Judge answer quality with an explicit local agent command:

```sh
bun run eval:judge -- --run local-smoke --judge-command "claude -p"
```

Generated eval runs live under `fixtures/skill-evals/runs/`. Generated demo runs live under `fixtures/runs/`. Both are ignored by git and excluded from the npm package.

## What Is Where

- `scanner.test.js` checks scanner behavior, including temp-file edge cases.
- `fixtures/scanner-cases.md` is the dense scanner regression fixture.
- `fixtures/scanner-cases.expected.json` is the exact scanner output expected for that fixture.
- `demo.test.js` checks the human demo input has intended matches and the processed demo is sealed.
- `fixtures/demo.md` is the human-readable before example.
- `fixtures/demo.processed.md` is the curated human-readable after reference.
- `test/scripts/run-demo-skill.js` copies `fixtures/demo.md` to `fixtures/runs/<run-id>/demo.md`, invokes a real LLM command with `SKILL.md`, and scans the generated output.
- `eval-fixtures.test.js` checks skill eval inputs and expected outputs stay paired.
- `eval-scripts.test.js` checks the eval prepare, verify, and judge scripts.
- `fixtures/skill-evals/input/` contains files the executor is allowed to see.
- `fixtures/skill-evals/expected/` contains judge-only ideal outputs.
- `fixtures/skill-evals/runs/` contains generated eval runs and should not be committed.

## How To Read Results

`bun run test` should be fully pass/fail. A failure means either code changed behavior or a fixture is out of date.

`bun run test:review` runs the test suite, invokes a real LLM against a generated copy of `demo.md`, and fails if the generated demo still has actionable comments. Review the generated output against `fixtures/demo.processed.md`; exact text does not need to match.

`bun run eval:verify` emits deterministic JSON. Treat this as a protocol smoke check, not the final semantic judge. The top-level `score` and each case score are partial-credit values from 0 to 1.

`bun run eval:judge` spawns the command passed through `--judge-command` or `MDAC_JUDGE_COMMAND`, passes the generated `runs/<run-id>/judge-prompt.md`, and writes `judge-result.json` when the judge returns valid JSON.

Useful statuses:

- `pass`: exact or near-exact expected behavior.
- `pass-with-notes`: usable output with meaningful differences from expected.
- `partial`: the agent did some work but missed important pieces.
- `fail`: setup or execution is substantially wrong.

The verifier script is a mechanical baseline. Use the LLM judge output plus direct markdown review for answer quality.

## Agent Notes

Do not point the executor at `fixtures/skill-evals/expected/`. The expected files are for the judge.

Do not paste this README into the executor chat. Paste only the generated `executor-prompt.md`.

The executor prompt generated by `eval:prepare` includes absolute local paths. That is intentional for local semi-manual runs.

If `eval:verify` says the `actual/` directory does not exist, run `eval:prepare` first or check the run id.

The scanner fixture intentionally includes a `kind: "done"` match. A `[!DONE]-` thread becomes actionable again when a human adds a follow-up after the latest `<!--mdac:eot-->`.
